## 1. Identity & Role Management

**BR-IDM-01 — Account creation paths are role-gated at registration.**  
A CEO (`active_role = CLIENT`, `client_subtype = CEO`) self-registers via `POST /auth/register`. A TECH_TEAM account can only be created by consuming a signed JWT handoff link (`client_subtype: TECH_TEAM`, `project_id`, 72-hour expiry) generated by a CEO during Stage 4 of elicitation. There is no self-registration path for TECH_TEAM. Admin accounts are DB-seeded at deploy only — there is no registration endpoint for ADMIN. `[APP]`

**BR-IDM-02 — TECH_TEAM account is permanently scoped to one project.**  
`tech_team_profiles.linked_project_id → projects.id` is set at registration (immediately if the project exists) and is immutable. A TECH_TEAM user cannot view, act on, or be invited to any project other than the one encoded in their handoff link. Route guards on all TECH_TEAM-facing endpoints validate this scope. `[APP]` `[DB: FK]`

**BR-IDM-03 — The handoff link expires after 72 hours.**  
A handoff link whose embedded JWT has `exp < now()` is rejected at `POST /auth/register/handoff` with 401 LINK_EXPIRED. The CEO must generate a new link. A single link can only be consumed once (single account creation). `[APP]`

**BR-IDM-04 — Dual-role accounts are supported; active_role controls the current context.**  
A single `users` row may hold `roles: ["CLIENT_CEO","EXPERT"]`. `active_role` determines which dashboard is rendered and which routes are permitted. Role switching reissues a JWT (`PUT /auth/switch-role`) without re-login and re-evaluates all guards against the new `active_role`. `[APP]`

**BR-IDM-05 — Self-exclusion is permanent regardless of active_role.**  
A dual-role user (CLIENT_CEO + EXPERT) is always excluded from being matched as an expert on any project where they are the `projects.client_id`. This exclusion is applied in the matching engine query (`WHERE expert.user_id != project.client_id`) and is enforced even when the user switches to their EXPERT role. `[APP]`

**BR-IDM-06 — A Scenario B self-technical CEO gains TECH_TEAM capabilities only for the specific project.**  
When `project.self_technical = true`, the CEO's JWT is updated with `self_technical_projects: [project_id]`. Route guards on TECH_TEAM-scoped endpoints check for this claim for the specific `project_id`. The CEO does not gain TECH_TEAM capabilities platform-wide. `[APP]`

**BR-IDM-07 — JWT payload is the single source of truth for in-request authorization.**  
Every protected route validates: `sub` (user_id), `active_role`, `client_subtype`, `roles[]`, and (where relevant) `self_technical_projects[]`. No additional DB lookup for role resolution is permitted on the hot path. `[APP]`

**BR-IDM-08 — A suspended account (`users.is_active = false`) cannot authenticate.**  
Login attempts for a suspended account return 403 ACCOUNT_SUSPENDED. Existing JWTs for a suspended account are invalidated on next verification. Only Admin can suspend/reactivate accounts. `[APP]`

**BR-IDM-09 — Password resets require a valid, unexpired token.**  
`GET /auth/verify-reset-token/:token` must be called by the FE on page load before displaying the reset form. If `password_reset_token_expires_at < now()` or token is null, 400 is returned. Resetting successfully clears the token and invalidates all sessions (`refresh_token_hash = null`). `[APP]`

**BR-IDM-10 — Logout invalidates the refresh token server-side.**  
`POST /auth/logout` clears `users.refresh_token_hash`. The FE must also clear localStorage tokens. Subsequent attempts to use the invalidated refresh token will be rejected. `[APP]`

## 2. Subscription & Feature Gates

**BR-SUB-01 — Client Pro gates all LLM-powered client features.**  
AI Elicitation Engine (F2), Expert Matching & Shortlist (F4), and Artifact B access (F5) all require `users.subscription_client_tier = 'pro'` AND `sub_client_expires_at > now()`. Attempting to access these routes without an active Pro subscription returns 403 `{ code: "SUBSCRIPTION_REQUIRED" }`. `[ROUTE]`

**BR-SUB-02 — Expert Pro gates expert LLM features and Tier 2+ bidding.**  
LLM portfolio evidence verification (Tier 2 upgrade), bidding on Tier 2–3 projects, AI Service Generator, and the earnings analytics dashboard all require `users.subscription_expert_tier = 'pro'` AND `sub_expert_expires_at > now()`. `[ROUTE]`

**BR-SUB-03 — Path B marketplace browsing and service purchasing are Free-tier accessible.**  
Any `active_role = CLIENT` user (CEO or TECH_TEAM) may browse service listings (`GET /services`) without a Pro subscription. CEO may purchase a SERVICE_PURCHASE or TECH_DISCOVERY engagement without a Pro subscription. TECH_TEAM cannot purchase — route guard requires `client_subtype = 'CEO'`. `[ROUTE]`

**BR-SUB-04 — Expert profile creation (Tier 1 Claimed seam declarations) is Free-tier for experts.**  
Any registered expert may create their taxonomy profile, declare domain depths, and add Tier 1 seam claims without Expert Pro. Tier 2 verification requires Pro. `[ROUTE]`

**BR-SUB-05 — Subscription activation requires `packageId` and deducts via internal wallet.**  
`POST /subscriptions/activate { activeRole, packageId }` requires a valid `packageId` fetched from `GET /config/subscription-packages`. The price is deducted from `wallets.available_balance` in an atomic DB transaction. There is no SePay VA created for subscription payments. `[APP]`

**BR-SUB-06 — Subscription activation is guarded by sequential checks.**  
Guard 1: if `subscription_{role}_tier = 'pro'` AND `sub_{role}_expires_at > now()` → 409 ALREADY_SUBSCRIBED. Guard 2: if `wallets.available_balance < price` → 422 INSUFFICIENT_BALANCE. Both guards run before any DB writes. A `subscription_purchase_logs` row is inserted on success. `[APP]`

**BR-SUB-07 — Subscription state lives directly on `users`, with history in `subscription_purchase_logs`.**  
Four columns on `users` represent current state: `subscription_client_tier`, `subscription_expert_tier`, `sub_client_expires_at`, `sub_expert_expires_at`. The audit trail of purchases is provided by the `subscription_purchase_logs` table. `[DB]`

**BR-SUB-08 — Active engagements are grandfathered when a subscription expires.**  
If `sub_{role}_expires_at` passes while an engagement is in progress, the engagement continues uninterrupted. However, new LLM-gated actions (e.g., starting a new elicitation session) are blocked until renewal. `[APP]`

**BR-SUB-09 — Expired subscriptions are auto-corrected in API responses.**  
`GET /subscriptions/status` checks `expires_at`. If expired, the API returns `subscriptionTier: 'free'` and `isExpired: true`. The DB column may remain `'pro'`, but the API enforces the downgrade. FE must trust the API `subscriptionTier` and not compute expiry locally. `[APP]`

## 3. Wallet & Internal Ledger

**BR-WAL-01 — Every wallet-holding user has exactly one wallet row created atomically at registration.**  
`INSERT INTO wallets (user_id, available_balance, locked_balance) VALUES (?, 0, 0)` is part of the same atomic DB transaction as `INSERT INTO users`. No wallet can exist without a parent `users` row, and no `users` row is created without a wallet. `[DB: FK CASCADE, atomic TX]`

**BR-WAL-02 — `available_balance` and `locked_balance` can never go below zero.**  
Both columns have `CHECK (available_balance >= 0)` and `CHECK (locked_balance >= 0)`. Any deduction that would produce a negative value is rejected at the DB level before the application-level guard even fires. `[DB: CHECK]`

**BR-WAL-03 — The `wallet_transactions` table is an immutable, append-only ledger.**  
No UPDATE or DELETE operation is ever performed on `wallet_transactions`. Every financial event generates a new INSERT. Reversals create a new row with a distinct `reference_id` (e.g., `'WD-CANCEL-{id}'`), never a modification. `[APP]`

**BR-WAL-04 — All money amounts are stored as BIGINT (VND integer, no decimals).**  
Financial calculations that produce fractional values are rounded at the application layer before storage. Platform fee is computed as `ROUND(amount * platform_fee_pct)`. `[DB: BIGINT]`

**BR-WAL-05 — The platform fee percentage is read from `platform_settings` at transaction time — never hardcoded.**  
`SELECT platform_fee_pct FROM platform_settings LIMIT 1` is issued inside every APPROVED ledger transaction. `platform_settings` is a singleton row seeded at deploy with `platform_fee_pct = 0.05`. The singleton is referenced by `platform_settings.platform_wallet_id` for the destination wallet. `[APP]`

**BR-WAL-06 — Every wallet deduction is atomic with its corresponding event.**  
ESCROW_LOCK, SUBSCRIPTION, and WITHDRAWAL deductions are performed inside the same DB transaction as the state change they enable (e.g., `milestones.state → FUNDED` and `wallets.available_balance -= amount` commit or roll back together). Partial commits are not possible. `[DB: atomic TX]`

**BR-WAL-07 — Nine transaction types cover all possible ledger events.**  
`wallet_transactions.transaction_type CHECK IN ('TOP_UP','SUBSCRIPTION','ESCROW_LOCK','ESCROW_RELEASE','PLATFORM_FEE','ESCROW_REFUND','ESCROW_SPLIT','WITHDRAWAL','WITHDRAWAL_REFUND')`. No other string values are accepted. `[DB: CHECK]`

**BR-WAL-08 — Dual-role users share a single wallet across both roles.**  
A user holding both CLIENT_CEO and EXPERT roles has one `wallets` row. Expert earnings credited via ESCROW_RELEASE can be used immediately to fund a milestone as CEO (same `available_balance`). `[DB: 1:1 users-wallets]`

## 4. SePay Payment Integration

**BR-PAY-01 — Every inbound IPN must pass HMAC signature verification before any DB operation.**  
`POST /webhooks/sepay/ipn` verifies the SePay HMAC signature using `SEPAY_SECRET_KEY` as the first action. Any request that fails verification is rejected with 401 and no DB writes occur. `[APP]`

**BR-PAY-02 — SePay IPN idempotency is enforced at the DB level.**  
`UNIQUE INDEX wallet_tx_idempotency ON wallet_transactions(wallet_id, reference_id) WHERE reference_id IS NOT NULL`. If a SePay IPN retry delivers the same `transfer_reference` (mapped to `reference_id`), the INSERT on `wallet_transactions` fails the unique constraint. The handler catches this, logs it, and returns HTTP 200 to prevent further retries — but takes no financial action. `[DB: UNIQUE INDEX]`

**BR-PAY-03 — NestJS always returns HTTP 200 synchronously to SePay before any async operations.**  
All IPN processing (ledger writes, state transitions, pay-gated doc releases) completes synchronously within the handler. SePay retries on any non-2xx response, so 200 must be returned even on previously-seen events. `[APP]`

**BR-PAY-04 — IPN branch routing is determined by `virtual_accounts.entity_type`.**  
The handler resolves `va_number → virtual_accounts.entity_type` and branches: WALLET_TOPUP (credit available_balance), MILESTONE (escrow lock + milestone FUNDED + pay-gated doc release), SERVICE (escrow lock + engagement ACTIVE). Only these three entity_type values are valid. `[APP]` `[DB: CHECK entity_type]`

**BR-PAY-05 — MILESTONE and SERVICE VAs enforce exact-amount transfers.**  
`virtual_accounts.fixed_amount` is set for all non-TOPUP VAs. SePay / the bank rejects transfers of any amount other than `fixed_amount`. An IPN that arrives with an amount mismatch vs. `va.fixed_amount` is rejected by the handler — no ledger write, 422 to SePay. `[APP]`

**BR-PAY-06 — WALLET_TOPUP VAs are permanent and accept any transfer amount.**  
WALLET_TOPUP VAs have `fixed_amount = NULL` and `expires_at = NULL`. They are created once at user registration and are never re-created or expired. The same VA number is used for all top-ups for a given user. `[APP]` `[DB: fixed_amount NULL for TOPUP]`

**BR-PAY-07 — MILESTONE VAs expire after 24 hours.**  
`virtual_accounts.expires_at = now() + 24 hours` is set on creation. An IPN arriving after expiry is rejected. The CEO must re-trigger VA creation (`PUT /milestones/:id/fund`) to generate a fresh VA. `[APP]`

**BR-PAY-08 — Expert bank account linking does NOT make a real SePay API call in MVP.**  
`POST /bank-hub/initiate-link` stores the `sepay_bank_account_xid` and `bank_account_holder_name` as placeholders for admin reference. The platform never handles real bank credentials in the MVP. `[APP]`

**BR-PAY-09 — Withdrawals do not trigger an automatic Chi Hộ API call.**  
When an expert requests a withdrawal (`POST /withdrawals`), a `withdrawal_requests` row is created in `PENDING` state and the wallet is debited atomically. No SePay Chi Hộ API is called. `[APP]`

**BR-PAY-10 — Admins manually complete or fail withdrawals.**  
Admin verifies the external bank transfer and calls `/admin/withdrawals/{id}/complete` (moves to `COMPLETED`) or `/admin/withdrawals/{id}/fail` (moves to `FAILED`, atomically restores wallet balance via `WITHDRAWAL_REFUND` ledger entry). `[APP]`

**BR-PAY-11 — Expert can cancel a pending withdrawal.**  
`DELETE /withdrawals/:id` checks `status = 'PENDING'`. If true, sets status to `CANCELLED` and atomically restores the wallet balance via `WITHDRAWAL_REFUND`. `[APP]`

**BR-PAY-12 — Expert cannot request withdrawal without a linked bank account.**  
`POST /withdrawals` has a guard: `users.sepay_bank_account_xid IS NOT NULL`. If NULL → 422 with redirect to Bank Hub link flow. `[ROUTE]`

## 5. AI Elicitation Engine

**BR-ELI-01 — Client Pro subscription is the entry gate for elicitation.**  
`POST /elicitation/sessions` is guarded by `users.subscription_client_tier = 'pro'` AND `sub_client_expires_at > now()`. 403 is returned before Stage 1 if the subscription is inactive. `[ROUTE]`

**BR-ELI-02 — The elicitation session is the only mechanism for creating a PROJECT_BASED project.**  
A `projects` row with `elicitation_session_id IS NOT NULL` can only be created by the Stage 5 synthesis engine. No direct `POST /projects` endpoint exists for project creation outside elicitation. `[APP]`

**BR-ELI-03 — Archetype is locked after Stage 2 and cannot be changed.**  
`elicitation_sessions.archetype` is set during Stage 2 and is immutable for the rest of the session. Re-entering a returned session resumes with the locked archetype. `[APP]`

**BR-ELI-04 — CEO cannot access Stage 4 directly without a self-technical flag.**  
`PUT /elicitation/sessions/:id/stage4` returns 403 for any request where `active_role = CLIENT/CEO` AND `project.self_technical = false` for that project. Only `client_subtype = TECH_TEAM` (via handoff link) or `client_subtype = CEO` with `self_technical = true` for that project may submit Stage 4 inputs. `[ROUTE]`

**BR-ELI-05 — Void detection injects mandatory milestones and gates completeness.**  
If the LLM detects an absent ground truth baseline (void) in Stage 1, it is recorded in `void_list_json`. If critical artifacts are flagged in `critical_artifacts_json` and not provided in Stage 4, the Stage 5 synthesis is capped at `completeness_score = 0.60` and specific sdlc_notices are added. `[APP]`

**BR-ELI-06 — Auto-publish quality gate requires `completeness_score >= 0.70`.**  
The gate runs in `POST /elicitation/sessions/{id}/stage5`. It checks: `completeness_score >= 0.70`. If it fails, the spec is set to `RETURNED_TO_CLIENT`. If it passes, the project is `PUBLISHED`. `[APP]`

**BR-ELI-07 — Quality gate failure is self-correcting — there is no admin approval queue.**  
On gate failure: `elicitation_sessions.state → 'RETURNED'`; no `projects` row is created; `INSERT INTO platform_decisions { decision_type: 'SPEC_AUTO_RETURN', advisory_note: {LLM-generated targeted void note} }`. The CEO re-enters at the specific failing void, not from Stage 1. `[APP]`

**BR-ELI-08 — A `projects` row is created only when the quality gate passes.**  
On gate pass: one atomic transaction creates the `projects` row (`state = 'PUBLISHED'`) with all JSONB columns, updates `elicitation_sessions.state = 'COMPLETED'`, and writes a `platform_decisions` row. If any part fails, the transaction rolls back and no partial state persists. `[DB: atomic TX]`

**BR-ELI-09 — Scenario A offers TECH_DISCOVERY; Scenario B grants CEO the Stage 4 form.**  
Scenario A (`scenario_type = 'SCENARIO_A'`): system offers (a) inject TECH_DISCOVERY as Milestone 0 or (b) purchase a TECH_DISCOVERY service first. Scenario B (`scenario_type = 'SCENARIO_B'`): `project.self_technical = true`; CEO completes Stage 4 form directly. The scenario type is written to `elicitation_sessions.scenario_type` and is immutable. `[APP]`

**BR-ELI-10 — An abandoned session preserves `current_stage` for resumption.**  
`elicitation_sessions.state = 'ABANDONED'` is set on navigation away. `current_stage` is preserved. A new session start by the same user resumes the existing session. There is no hard delete of an in-progress session. `[APP]`

**BR-ELI-11 — The JSONB footprint columns on `projects` replace three cut tables.**  
`required_seams_json`, `required_domains_json`, `milestone_framework_json`, `artifact_a_json`, and `artifact_b_json` are all columns on the single `projects` row. The `capability_footprints`, `artifact_a`, and `artifact_b` tables from the full specification do not exist in the MVP schema. `[DB: schema]`

**BR-ELI-12 — Stage 4 drafts can be auto-saved without triggering LLM.**  
`PATCH /elicitation/sessions/:id/stage4-draft` updates `stage4_draft_json` instantly. This is used for FE auto-save (every 30s) to prevent data loss. `[APP]`

**BR-ELI-13 — Missing critical artifacts in Stage 4 do not hard-block submission.**  
Stage 4 returns `{ session, missingArtifacts }`. If `missingArtifacts` is non-empty, the FE shows a warning modal, but the CEO can proceed. The AI will generate a generic spec with capped completeness. `[APP]`

## 6. Expert Profile & Seam Verification

**BR-VER-01 — Every seam claim starts at Tier 1 (CLAIMED) on self-declaration.**  
`INSERT INTO expert_seam_claims { verification_tier: 'CLAIMED', submission_count: 0, locked_until: NULL }`. Tier 1 carries a composite score confidence weight of 0.5. `[DB]` `[§0.4]`

**BR-VER-02 — Each expert has at most one claim row per seam.**  
`UNIQUE (expert_id, seam_code)` on `expert_seam_claims`. A Tier 2 upgrade is an UPDATE on the existing row, not a new INSERT. `[DB: UNIQUE]`

**BR-VER-03 — Tier 2 upgrade requires Expert Pro AND LLM confidence ≥ 0.85 AND all required signal types.**  
`POST /portfolio-submissions` is guarded by `subscription_expert_tier = 'pro'`. FastAPI LLM evaluation produces `{ confidence_score, passed_boolean }`. If `confidence_score >= 0.85 AND passed_boolean = true`: `UPDATE expert_seam_claims SET verification_tier = 'EVIDENCE_BACKED'`. Tier 2 confidence weight = 1.0. `[ROUTE]` `[APP]` `[§0.4]`

**BR-VER-04 — Tier 2 upgrade is fully automatic — no admin approval.**  
On pass: `INSERT INTO platform_decisions { decision_type: 'SEAM_TIER_UPGRADE' }`. No admin review step exists. `[APP]`

**BR-VER-05 — A rejection produces a targeted LLM advisory note identifying missing signals.**  
On fail: `INSERT INTO platform_decisions { decision_type: 'PORTFOLIO_EVAL', decision: 'REJECTED', advisory_note: {gap advisory} }`. The advisory names which required signal types were found vs. missing. The expert sees this advisory in their dashboard. `[APP]`

**BR-VER-06 — An expert may attempt Tier 2 verification at most 5 times per seam per 30-day window.**  
`expert_seam_claims.submission_count` increments on each attempt. On the 5th failure: `UPDATE expert_seam_claims SET locked_until = now() + 30 days`. While `locked_until > now()`, new submissions for that seam return 429 TOO_MANY_ATTEMPTS. `[APP]`

**BR-VER-07 — Tier 3 and Tier 4 do not exist in the MVP.**  
`expert_seam_claims.verification_tier CHECK IN ('CLAIMED','EVIDENCE_BACKED')`. Scenario assessments (`scenario_assessments`, `scenario_responses` tables) and post-engagement outcome signals (`expert_seam_outcome_signals` table) are deferred to Phase 2. `[DB: CHECK]`

**BR-VER-08 — Stack tags are stored as `stack_tags_json JSONB` on `expert_profiles` — not a separate table.**  
`expert_stack_tags` table is not built in the MVP. Format: `["Python","Kafka","Go"]`. Archetype history is similarly stored as `archetype_history_json JSONB` on `expert_profiles`: `[{archetype_code, tier, self_declared: true}]`. `[DB: schema]`

**BR-VER-09 — Domain depth levels are bounded.**  
`expert_domain_depths.depth_level CHECK IN ('SURFACE','OPERATIONAL','DEEP')`. Each expert has at most one depth row per domain (`UNIQUE(expert_id, domain_code)`). `[DB: CHECK, UNIQUE]`

## 7. Composite Matching Engine

**BR-MAT-01 — Matching fires automatically when a project is published.**  
On `projects.state → 'PUBLISHED'`, NestJS triggers the matching engine via FastAPI `/llm/matching`. Results are cached in `project_shortlist_cache`. `[APP]`

**BR-MAT-02 — The composite score uses exactly five weighted components.**  
Per §0.5: Seam alignment 40%, Domain depth coverage 25%, Portfolio quality 20%, Archetype history 10%, Engagement model fit 5%. Weights sum to 100%. `[APP]` `[§0.5]`

**BR-MAT-03 — Seam alignment uses the two-tier confidence weighting system.**  
For each required seam: Tier 2 (EVIDENCE_BACKED) = 1.0 confidence × criticality weight; Tier 1 (CLAIMED) = 0.5 confidence × criticality weight. A missing seam contributes 0.0. `[APP]` `[§0.4]` `[§0.5]`

**BR-MAT-04 — The hard gate excludes experts with a claimed-to-verified ratio > 4:1 among required seams.**  
Before composite scoring, the engine calculates `COUNT(seam_claims WHERE verification_tier='CLAIMED' AND seam_code IN required_seams) / COUNT(seam_claims WHERE verification_tier='EVIDENCE_BACKED' AND seam_code IN required_seams)`. If this ratio exceeds 4:1, the expert is excluded entirely. The gate only applies if evidence_backed_count > 0. `[APP]`

**BR-MAT-05 — Clients see match strength labels, not numeric scores.**  
Strength label thresholds per §0.5: STRONG_MATCH (composite ≥ 0.85), GOOD_MATCH (0.70–0.85), POSSIBLE_MATCH (0.55–0.70), WEAK_MATCH (< 0.55). Scores below 0.55 are still returned but labeled WEAK_MATCH. `[APP]` `[§0.5]`

**BR-MAT-06 — Seam gap map uses three colors: Green (Tier 2), Amber (Tier 1), Red (Absent).**  
Green (EVIDENCE_BACKED), Amber (CLAIMED), Red (Missing). For each required seam in the project footprint, each matched expert receives one of these three gap map entries. `[APP]` `[§0.4]`

**BR-MAT-07 — CEO can force re-score via `refresh=true`.**  
`GET /matching/{projectId}/shortlist?refresh=true` bypasses the `project_shortlist_cache` and calls FastAPI `/llm/matching` again with the latest expert profiles. `[APP]`

**BR-MAT-08 — Matching requires Client Pro subscription.**  
`GET /matching/{projectId}/shortlist` is guarded by `subscription_client_tier = 'pro'`. Free-tier CEOs cannot see shortlists. `[ROUTE]`

**BR-MAT-09 — Self-exclusion is enforced in the matching query.**  
The expert pool query always includes `WHERE expert.user_id != projects.client_id` (and the broader `NOT IN project member list`). This prevents a dual-role user from appearing as an expert candidate on their own project. `[APP]`

## 8. Information Release & IP Escrow

**BR-ART-01 — Artifact A is publicly visible to all matched experts pre-bid.**  
`projects.artifact_a_json` is returned by `GET /projects/{id}/artifact-a` to any expert in the match shortlist for that project. No connection or NDA is required to view Artifact A. `[ROUTE]`

**BR-ART-02 — Artifact B has a four-condition route guard.**  
`GET /projects/{id}/artifact-b` (FastAPI) returns `artifact_b_json` only when ALL FOUR conditions are simultaneously true: (a) `engagements.state IN ('CONNECTED', 'ACTIVE')`, (b) `capability_bids.state IN ('TECH_APPROVED', 'CEO_REVIEW', 'SELECTED')`, (c) `engagements.expert_nda_accepted_at IS NOT NULL`, (d) `engagements.client_nda_accepted_at IS NOT NULL`. `[ROUTE]`

**BR-ART-03 — CEO is permanently excluded from Artifact B at the route level.**  
Regardless of engagement state, NDA status, subscription, or role, a request where `requester.active_role = CLIENT` AND `requester.client_subtype = 'CEO'` returns 403. There is no override path. `[ROUTE]`

**BR-ART-04 — Both NDA click-throughs are required before `engagement.state → CONNECTED`.**  
`engagements.client_nda_accepted_at` and `engagements.expert_nda_accepted_at` must both be non-NULL before the `CONNECTED` state transition is permitted. Neither party can unilaterally advance the state. `[APP]`

**BR-ART-05 — Pay-gated documents are staged before funding and released atomically on funding.**  
`paygated_documents.release_state = 'STAGED'` is set when the expert uploads a document tagged to a milestone. `release_state → 'RELEASED'` occurs inside the same DB transaction that processes the MILESTONE IPN (`milestones.state → 'FUNDED'`, `escrow_accounts { status: 'HELD' }`, ledger ESCROW_LOCK). Release cannot be triggered independently of payment. `[DB: atomic TX]`

**BR-ART-06 — Pay-gated documents are accessible to TECH_TEAM and EXPERT only — CEO is permanently excluded.**  
`GET /milestones/{id}/paygated-docs` is guarded by `active_role IN ('TECH_TEAM', 'EXPERT')` AND `release_state = 'RELEASED'`. CEO requests return 403. `[ROUTE]`

**BR-ART-07 — NDA click-through also triggers the Bank Hub prompt for experts without a linked account.**  
When `engagements.state → 'CONNECTED'` and `expert.sepay_bank_account_xid IS NULL`, the system displays a non-blocking prompt for the expert to link their bank account. Connection is not blocked by missing bank account, but withdrawal capability (BR-PAY-12) is. `[APP]`

## 9. Capability Bid Lifecycle

**BR-BID-01 — Exactly one bid is permitted per engagement.**  
`capability_bids.engagement_id UNIQUE` is a DB-level constraint. A second `INSERT INTO capability_bids` for the same `engagement_id` fails with a unique violation. `[DB: UNIQUE]`

**BR-BID-02 — SERVICE_PURCHASE and TECH_DISCOVERY engagements never have a bid.**  
The `capability_bids` table has no row for Path B/C engagements. These engagement types are created directly in the IPN handler and skip the entire bid lifecycle. `[APP]`

**BR-BID-03 — All three bid components are required at SUBMITTED transition.**  
`footprint_alignment_json` (JSONB), `approach_summary` (TEXT), and `conditional_pricing_json` (JSONB) must all be non-NULL when the bid transitions to `SUBMITTED`. If any component is NULL → 422 with the name of the missing component. `[APP]`

**BR-BID-04 — TECH_TEAM must approve before CEO can review — enforced at the route level.**  
`PUT /bids/{id}/ceo-decision` reads `capability_bids.tech_status`. If `tech_status != 'APPROVED'` → 422 `"Tech review not complete"`. No exception exists. `[ROUTE]`

**BR-BID-05 — Bid revisions are mutable in-place updates on the single row — no version table.**  
When `tech_status = 'REVISION_REQUESTED'`, the expert edits any bid component directly: `UPDATE capability_bids SET {component} = ?, tech_status = 'PENDING', version_number += 1`. Previous component content is not preserved as a separate snapshot. `[APP]` `[DB: no bid_versions table]`

**BR-BID-06 — `tech_feedback` is a single TEXT column — not a table.**  
TECH_TEAM writes its revision feedback to `capability_bids.tech_feedback`. The expert reads this single text value. There is no structured per-component revision tracking (that was `bid_revision_requests` in the full spec). `[DB: column]`

**BR-BID-07 — Counter-offer is a single BIGINT value, written once.**  
`capability_bids.negotiated_price_vnd` is the only price negotiation mechanism in the MVP. It can only be set once (no overwrite once non-NULL). There is no per-milestone breakdown, no round tracking, and no formal expert counter-counter-offer round. `[APP]`

**BR-BID-08 — Pre-bid technical questions use the messages channel — no Surface A table.**  
The `spec_clarifications` table from the full specification is not built. All expert questions before bid submission go to the `messages` table associated with the project context (`project_id` set, `engagement_id` null). No status tracking (OPEN/ANSWERED/RESOLVED) exists per question. `[APP]` `[DB: no spec_clarifications table]`

**BR-BID-09 — Expert Pro is required to bid on Tier 2–3 projects.**  
`POST /bids` checks `project.tier` against `expert.subscription_expert_tier`. Tier 2–3 projects require `subscription_expert_tier = 'pro'`. Tier 1 projects can be bid on without Expert Pro. `[ROUTE]` `[§0.9]`

**BR-BID-10 — ceo_status and tech_status are independent columns with separate guards.**  
CEO setting `ceo_status = 'APPROVED'` or `'DECLINED'` requires only `tech_status = 'APPROVED'`. There is no Surface D override mechanism for CEO to bypass a REVISION_REQUESTED tech_status (the full spec's `bid_conflict_overrides` table is not built). `[APP]`

**BR-BID-11 — Seam and Domain codes in bid DTOs are dynamic strings.**  
`code` fields in `footprint_alignment_json` accept any non-empty string. Seam codes MUST use the `↔` Unicode arrow. The old `<->` ASCII format is rejected. Valid codes are validated against the CMS config tables at the service layer. `[APP]`

## 10. Milestone & Deliverable Workflow

**BR-MIL-01 — Milestone number is unique per engagement.**  
`UNIQUE (engagement_id, milestone_number)` is a DB-level constraint. Auto-incrementing within an engagement is enforced at the application layer. `[DB: UNIQUE]`

**BR-MIL-02 — A per-milestone VA with enforced fixed_amount is created when CEO clicks "Fund."**  
`PUT /milestones/{id}/fund` creates `INSERT INTO virtual_accounts { entity_type: 'MILESTONE', fixed_amount: payment_amount_vnd, expires_at: now() + 24h }`. The bank rejects any transfer of a different amount. `[APP]` `[BR-PAY-05]`

**BR-MIL-03 — Milestone VA expires after 24 hours.**  
If the VA expires before payment, the CEO must re-click "Fund" to create a new VA with a fresh 24-hour expiry. The expired VA's `status → 'EXPIRED'`. `[APP]`

**BR-MIL-04 — `FUNDED → IN_PROGRESS` is an automatic system transition, not a user action.**  
Immediately after `milestones.state → 'FUNDED'` within the IPN handler, the system auto-advances to `state = 'IN_PROGRESS'`. Expert begins billable work only after escrow is confirmed. `[APP]`

**BR-MIL-05 — The first milestone funded on any engagement advances `engagement.state → 'ACTIVE'`.**  
Within the same IPN transaction that funds Milestone 1: `UPDATE engagements SET state = 'ACTIVE'`. Subsequent milestones do not trigger this again. `[APP]`

**BR-MIL-06 — Sign-off authority is set at milestone creation and is immutable.**  
`milestones.sign_off_authority CHECK IN ('TECH_TEAM','CEO','JOINT')`. Once set, this field cannot be changed. `[DB: CHECK]`

**BR-MIL-07 — JOINT milestones require both TECH_TEAM and CEO verification before APPROVED.**  
For `sign_off_authority = 'JOINT'`: NestJS checks that `verified_at` is set on all required criteria for both verifier roles. The APPROVED transition is blocked until both sets are complete. `[APP]`

**BR-MIL-08 — Platform fee is read from `platform_settings` at APPROVED transaction time.**  
`SELECT platform_fee_pct FROM platform_settings LIMIT 1` is issued inside the APPROVED atomic transaction. The fee is never hardcoded. Three `wallet_transactions` rows are written: ESCROW_RELEASE (client locked reduced), PLATFORM_FEE (platform wallet credited), and expert credit (ESCROW_RELEASE). `[APP]` `[BR-WAL-05]`

**BR-MIL-09 — Withdrawal request for milestone release is created on APPROVED.**  
The `withdrawal_requests` row is created (`type: 'MILESTONE_RELEASE'`, `status: 'PENDING'`) inside the APPROVED transaction. Admin must manually complete it via `/admin/withdrawals/{id}/complete`. Milestone state advances to `RELEASED` upon admin completion. `[APP]`

**BR-MIL-10 — Last milestone RELEASED closes the engagement.**  
When `withdrawal_requests.status → 'COMPLETED'` for the final milestone's `MILESTONE_RELEASE`: `UPDATE engagements SET state = 'CLOSED'`. Reviews become available at this point. `[APP]`

**BR-MIL-11 — Sprint tracking (Layer 3) does not exist in the MVP.**  
`milestone_sprints`, `sprint_status_updates`, `addon_phase_requests` tables are not built. There are no sprint status updates, no SCOPE_EVOLUTION triggers, and no Add-On Phase Protocol. If scope changes during an engagement, the CEO manually creates a new milestone. `[DB: no tables]`

**BR-MIL-12 — Milestone chat history is persisted server-side.**  
`POST /projects/:id/milestone-chat` stores messages in `milestone_chat_sessions.messagesJson`. The FE only needs to store the `chatSessionId` in state, never the full message array. `[DB: schema]`

## 11. Acceptance Criteria & DoD Checklist

**BR-ACC-01 — LLM criterion quality gate runs on every criterion save.**  
`POST /criteria/{milestoneId}` calls FastAPI `/llm/criterion-check { criterion_text, archetype_name, milestone_context }`. FastAPI returns `{ is_subjective, suggestions, severity, context_note }`. If subjective language is detected: flags are returned to the submitter with suggestions. `INSERT INTO platform_decisions { decision_type: 'CRITERION_QUALITY_GATE' }`. The criterion is saved regardless — the gate is advisory, not blocking, but the flags are surfaced to the user. `[APP]`

**BR-ACC-02 — Criterion verification is role-gated by `verified_by_role`.**  
`PUT /criteria/{id}/verify` checks `active_role` and `client_subtype` against `acceptance_criteria.verified_by_role`: TECH_TEAM criteria require `client_subtype = TECH_TEAM`; CEO criteria require `client_subtype = CEO`. A role mismatch returns 403. `[ROUTE]`

**BR-ACC-03 — `revision_note` on `acceptance_criteria` replaces the `revision_requests` table.**  
When a reviewer rejects a criterion, they write `acceptance_criteria.revision_note` in-place. This transitions the milestone to `IN_REVISION`. There is no separate `revision_requests` table in the MVP. `[DB: column, no revision_requests table]`

**BR-ACC-04 — APPROVED transition guard fires when all required criteria are verified.**  
After each `PUT /criteria/{id}/verify`, NestJS runs: `SELECT COUNT(*) FROM acceptance_criteria WHERE milestone_id = ? AND is_required = true AND verified_at IS NULL`. If 0 → atomic APPROVED ledger release fires. If > 0 → state remains as-is (no error — just not yet complete). `[APP]`

**BR-DOD-01 — A required DoD item can never be set to NOT_APPLICABLE.**  
`CONSTRAINT dod_required_cannot_be_na CHECK (NOT (is_required = TRUE AND status = 'NOT_APPLICABLE'))` is a DB-level check constraint. Any UPDATE attempting to mark a required item NOT_APPLICABLE will fail with a constraint violation before reaching the application layer. `[DB: CHECK]`

**BR-DOD-02 — SUBMITTED transition is blocked by any incomplete required DoD item.**  
`POST /milestones/{id}/submit` runs: `SELECT id, item_description FROM milestone_dod_items WHERE milestone_id = ? AND is_required = true AND status != 'COMPLETED'`. If any rows returned → 422 `DOD_INCOMPLETE` with the list of blocking items. This guard runs before any milestone state write. `[ROUTE]`

**BR-DOD-03 — Completion note is mandatory for required items marked COMPLETED.**  
Application-level validation on `PUT /milestones/{id}/dod/{itemId}`: if `is_required = true` AND `status = 'COMPLETED'` AND `completion_note` is NULL or empty → 422. `[APP]`

**BR-DOD-04 — NOT_APPLICABLE note is mandatory for non-required items marked NOT_APPLICABLE.**  
Application-level validation: if `status = 'NOT_APPLICABLE'` AND `not_applicable_note` is NULL or empty → 422. `[APP]`

**BR-DOD-05 — DoD visibility is role-restricted.**  
TECH_TEAM can read DoD items but cannot write them. CEO cannot read DoD items at all (route guard: 404/403 for CEO). Only Expert can create and update DoD items. `[ROUTE]`

**BR-DOD-06 — There are no DoD item comments in the MVP.**  
DoD-related discussion between expert and TECH_TEAM happens in the engagement messages channel. No `dod_item_comments` table exists. `[DB: no table]`

## 12. Escrow Operations

**BR-ESC-01 — Every escrow account has exactly one parent — either a milestone or an engagement, never both.**  
`CONSTRAINT escrow_has_one_parent CHECK ((milestone_id IS NOT NULL AND engagement_id IS NULL) OR (milestone_id IS NULL AND engagement_id IS NOT NULL))`. Violation fails at the DB level. `[DB: CHECK]`

**BR-ESC-02 — Path A (PROJECT_BASED) escrow is linked to `milestone_id`.**  
For each funded milestone, a separate `escrow_accounts` row is created with `milestone_id = ? AND engagement_id = NULL`. `[DB: partial UNIQUE INDEX escrow_milestone_unique]`

**BR-ESC-03 — Path B/C (SERVICE_PURCHASE, TECH_DISCOVERY) escrow is linked to `engagement_id`.**  
One `escrow_accounts` row per service purchase engagement, with `engagement_id = ? AND milestone_id = NULL`. `[DB: partial UNIQUE INDEX escrow_engagement_unique]`

**BR-ESC-04 — One escrow per milestone (Path A) and one escrow per engagement (Path B/C) are enforced by partial unique indexes.**  
`CREATE UNIQUE INDEX escrow_milestone_unique ON escrow_accounts(milestone_id) WHERE milestone_id IS NOT NULL` and `CREATE UNIQUE INDEX escrow_engagement_unique ON escrow_accounts(engagement_id) WHERE engagement_id IS NOT NULL`. `[DB: UNIQUE INDEX]`

**BR-ESC-05 — Escrow freeze on dispute is atomic and immediate.**  
`disputes.escrow_account_id` is a direct FK (not resolved through milestone or engagement chain). When a dispute is filed, `UPDATE escrow_accounts SET status = 'FROZEN'` runs in the same atomic transaction as `INSERT INTO disputes`. This enables O(1) freeze. `[DB: FK, atomic TX]`

**BR-ESC-06 — A frozen escrow cannot be released, refunded, or split until the dispute is resolved.**  
Application-level guard on all escrow-modifying operations checks `escrow_accounts.status != 'FROZEN'`. Any attempt to modify a frozen escrow outside the dispute resolution path is rejected. `[APP]`

**BR-ESC-07 — Escrow status transitions are one-way per event.**  
`escrow_accounts.status CHECK IN ('HELD','RELEASED','FROZEN','REFUNDED','SPLIT')`. Each status is a terminal outcome except FROZEN, which transitions to one of RELEASED/REFUNDED/SPLIT after resolution. HELD is the only initial status on creation. `[DB: CHECK]`

## 13. Dispute Resolution

**BR-DIS-01 — Disputes must be filed against a specific acceptance criterion.**  
`disputes.criterion_id NOT NULL REFERENCES acceptance_criteria(id)`. A dispute without a criterion anchor is invalid and rejected at the route level with 422. `[DB: NOT NULL FK]`

**BR-DIS-02 — Disputes also require a direct escrow reference.**  
`disputes.escrow_account_id NOT NULL REFERENCES escrow_accounts(id)`. This direct FK is mandatory — dispute filing cannot proceed without an identifiable escrow to freeze. `[DB: NOT NULL FK]`

**BR-DIS-03 — Any transactional role in the engagement may file a dispute.**  
CEO, TECH_TEAM, and Expert may all file disputes on a milestone in `SUBMITTED` or `IN_REVISION` state. Admin cannot file disputes. `[ROUTE]`

**BR-DIS-04 — Layer 1 LLM evaluation fires automatically after dispute filing.**  
`disputes.state` advances from `PENDING` to `LAYER_1_EVAL` as part of the dispute creation flow. NestJS calls FastAPI `/llm/dispute-eval { criterion_text, deliverable_description, project_archetype, milestone_context, prior_revision_count }`. No manual trigger is needed. `[APP]`

**BR-DIS-05 — LLM confidence threshold ≥ 0.80 triggers automatic resolution.**  
If `llm_confidence >= 0.80`: `disputes.state → 'AUTO_RESOLVED'`. Ledger is distributed per the LLM finding (expert wins = ESCROW_RELEASE chain; client wins = ESCROW_REFUND). `INSERT INTO platform_decisions { decision_type: 'DISPUTE_L1_EVAL', decision: 'AUTO_RESOLVED' }`. `[APP]`

**BR-DIS-06 — LLM confidence < 0.80 routes to Admin for manual resolution.**  
`disputes.state → 'MANUAL_REVIEW'`. Admin sees the dispute in the Dispute Monitor. Admin chooses one of three buttons. There is no automated resolution path at confidence < 0.80. `[APP]`

**BR-DIS-07 — Admin has exactly three resolution options for MANUAL_REVIEW disputes.**  
"Release to Expert" → ESCROW_RELEASE chain (same as happy-path APPROVED); "Refund to Client" → ESCROW_REFUND; "Split 50/50" → ESCROW_SPLIT (50% to each wallet). Each triggers the corresponding atomic ledger operation. `[APP]`

**BR-DIS-08 — Milestone state advances to APPROVED regardless of dispute outcome.**  
Whether the escrow is RELEASED, REFUNDED, or SPLIT, `milestones.state → 'APPROVED'` in the same transaction. The contractual lifecycle is closed. `[DB: atomic TX]`

**BR-DIS-09 — Layer 2 (48-hour cooling window) and Layer 3 (automated 50/50 split) do not exist in the MVP.**  
The full 3-layer dispute system is deferred to Phase 2. MVP has Layer 1 (LLM auto-resolve) and Layer 2 (admin manual button). `[DB: no dispute_resolution_reports table]`

**BR-DIS-10 — Additional evidence can be submitted after filing.**  
`POST /disputes/:id/evidence` allows filer to append `evidence_description` and `file_urls` to an open dispute (`LAYER_1_EVAL` or `MANUAL_REVIEW`). Stored as a `platform_decisions` row for audit. `[APP]`

**BR-DIS-11 — Disputes can be withdrawn by the filer before resolution.**  
`PUT /disputes/:id/withdraw` sets state to `WITHDRAWN` if the dispute is still in `LAYER_1_EVAL` or `MANUAL_REVIEW`. Escrow is unfrozen and returns to HELD status. `[APP]`

## 14. Messaging & Notifications

**BR-MSG-01 — Messages can be associated with either an engagement or a project.**  
The `messages` table has two nullable FKs: `engagement_id` and `project_id`. Exactly one must be non-null per row. Pre-bid Q&A uses `project_id`; post-connection chat uses `engagement_id`. `[DB: schema]`

**BR-MSG-02 — Admin may read all engagement threads but cannot send messages.**  
Route guard for `POST /engagements/{id}/messages` (WebSocket `message:send`) rejects `active_role = 'ADMIN'` with 403. Admin may issue `GET /engagements/{id}/messages` for dispute audit purposes. `[ROUTE]`

**BR-MSG-03 — Pre-bid questions use the messages channel — no spec_clarifications table.**  
Before an engagement exists, expert pre-bid questions are associated with the project context (project-level messages prior to bid submission). There is no OPEN/ANSWERED/RESOLVED status tracking per question in the MVP. `[APP]` `[DB: no spec_clarifications table]`

**BR-MSG-04 — Unread badge is computed on-demand — not stored as a counter.**  
Unread count for a given user on a given engagement is `COUNT(*) FROM messages m WHERE m.engagement_id = ? AND NOT EXISTS (SELECT 1 FROM message_reads mr WHERE mr.message_id = m.id AND mr.user_id = ?)`. `[APP]`

**BR-MSG-05 — A user cannot have duplicate read records for the same message.**  
`UNIQUE (message_id, user_id)` on `message_reads`. `[DB: UNIQUE]`

**BR-MSG-06 — Each message supports one file attachment in the MVP.**  
`messages.attachment_url TEXT NULL` — a single URL. Multiple-attachment messages require multiple messages. `[DB: column]`

**BR-MSG-07 — Socket.io room key is `user_id` for notifications, `engagement_id` for chat.**  
All real-time messaging is pub/sub scoped to the engagement. Notifications are scoped to the user. Messages are persisted to PostgreSQL before Socket.io broadcast. Loss of the Socket.io connection does not mean message loss. `[APP]`

**BR-NOT-01 — Notifications are both WebSocket-emitted and DB-persisted.**  
When a notification event fires (e.g., bid submitted), NestJS emits a `notification:generic` WebSocket event AND inserts a row into the `notifications` table. This ensures notifications survive page refresh. `[APP]` `[DB: schema]`

**BR-NOT-02 — Unread notification count is fetched via REST on page load.**  
`GET /notifications/me/unread-count` must be called on app mount to sync the nav badge. The WebSocket event should trigger a refetch, but the REST call is the source of truth for initial state. `[APP]`

## 15. Reviews & Reputation

**BR-REV-01 — Reviews may only be submitted after `engagements.state = 'CLOSED'`.**  
Route guard: `GET /engagements/{id}` is checked for `state = 'CLOSED'` before the review form is presented. Submission attempts on open engagements return 422. `[ROUTE]`

**BR-REV-02 — Exactly one review per engagement per reviewer.**  
`UNIQUE (engagement_id, reviewer_id)` on `reviews`. A second submission from the same reviewer for the same engagement fails the unique constraint. `[DB: UNIQUE]`

**BR-REV-03 — Three role-specific review forms write to the same `reviews` table.**  
`reviews.reviewer_role CHECK IN ('CEO','TECH_TEAM','EXPERT')` distinguishes the form context. CEO form: overall rating + comment. TECH_TEAM form: structured seam signals + rating. Expert form: engagement quality + client rating. `[DB: CHECK]`

**BR-REV-04 — `structured_signals_json` on `reviews` is informational only in MVP.**  
TECH_TEAM's seam performance signals are stored in `reviews.structured_signals_json` (JSONB) but do NOT trigger automatic seam tier upgrades. The `expert_seam_outcome_signals` table and Tier 4 signal accumulation are deferred to Phase 2. `[APP]` `[DB: no expert_seam_outcome_signals table]`

**BR-REV-05 — `reviews.structured_signals_json` absorbs the cut `outcome_signals` table.**  
The full spec's `outcome_signals` table is not built. Its data (seam code, signal type, seam role) is embedded as JSONB in the `reviews` row. `[DB: schema]`

**BR-REV-06 — Reputation aggregates are computed from `reviews` data.**  
Expert public reputation display: `AVG(rating)`, `COUNT(DISTINCT engagement_id)`, completion rate. These are computed at query time, not stored as counters. `[APP]`

## 16. Admin Governance & CMS

**BR-ADM-01 — Admin has specific write actions across the platform.**  
(a) Emergency spec pull-back: `PUT /admin/projects/{id}/suspend` → `projects.state = 'SUSPENDED'`. (b) Account suspension/reactivation: `PUT /admin/users/{id}/suspend` / `reactivate`. (c) Dispute resolution: `PUT /admin/disputes/{id}/resolve` → ledger + `disputes.state = 'RESOLVED'`. (d) Withdrawal processing: `PUT /admin/withdrawals/{id}/complete` / `fail`. (e) CMS management: Domains, Seams, Archetypes, Probes, Void Codes, Prompt Templates, Subscription Packages. `[APP]`

**BR-ADM-02 — Emergency spec pull-back is valid only for PUBLISHED projects.**  
`PUT /admin/projects/{id}/suspend` checks `projects.state = 'PUBLISHED'`. Attempting to pull back a DRAFT, RETURNED_TO_CLIENT, or already-SUSPENDED project returns 422. `[ROUTE]`

**BR-ADM-03 — Admin never touches money in normal operations.**  
The only financial writes Admin can perform are through the dispute resolution buttons (BR-DIS-07) and withdrawal completion/failure (BR-PAY-10). All routine financial flows (IPN handling, ledger entries) are fully automated. `[APP]`

**BR-ADM-04 — `platform_decisions` is written by FastAPI/NestJS and is never updated after insert.**  
This table is a pure audit log. Admins may read it for dashboards. Writes occur one row per LLM/AI decision event. No UPDATE or DELETE ever targets this table. `[APP]`

**BR-ADM-05 — Admin dashboard is monitoring-only with distinct module types.**  
Platform Integrity Monitor (reads `platform_decisions`), Dispute Monitor (reads `disputes` + MANUAL_REVIEW write), Account Management (reads `users`, writes `is_active`), Transaction Ledger (reads `wallet_transactions`, `escrow_accounts`, `withdrawal_requests`), Analytics (computed aggregates). `[APP]`

**BR-ADM-06 — CMS taxonomy changes take effect immediately.**  
Changes to Domains, Seams, Archetypes, Void Codes, and Probe Questions via `/admin/config/*` are reflected instantly in `/config/*` endpoints. No AI service restart is needed. `[APP]`

**BR-ADM-07 — Prompt template changes take effect within 60 seconds.**  
Admin updates to `prompt_templates` via `/admin/prompts/:stage` are fetched by FastAPI with a 60s TTL cache. Changes take effect within 1 minute without a service restart. `[APP]`

**BR-ADM-08 — Hard-deleting a subscription package is blocked if purchase history exists.**  
`DELETE /admin/subscriptions/packages/:id` returns 422 if any `subscription_purchase_logs` reference the package. Admin must use `PUT` with `isActive: false` to deactivate instead. `[APP]`

## 17. Database Integrity & Structural Invariants

**BR-INT-01 — All primary keys are UUID generated by `gen_random_uuid()`.**  
Requires `CREATE EXTENSION IF NOT EXISTS "pgcrypto"`. No auto-increment integer IDs. `[DB]`

**BR-INT-02 — All timestamps are `TIMESTAMPTZ` (UTC).**  
No `TIMESTAMP WITHOUT TIME ZONE` columns exist. All application-layer computations involving time use UTC. `[DB]`

**BR-INT-03 — All money is `BIGINT` (VND integer, no decimals).**  
`wallet_transactions.amount CHECK (amount > 0)` — always positive; direction is encoded in `transaction_type`. `[DB: CHECK]`

**BR-INT-04 — Engagement type FK consistency is enforced at the DB level.**  
`CONSTRAINT engagement_type_fk CHECK ((type = 'PROJECT_BASED' AND project_id IS NOT NULL AND service_id IS NULL) OR (type IN ('SERVICE_PURCHASE','TECH_DISCOVERY') AND project_id IS NULL AND service_id IS NOT NULL))`. `[DB: CHECK]`

**BR-INT-05 — Engagement type is immutable after creation.**  
There is no UPDATE path that changes `engagements.type`. Once created, the engagement type is permanent. `[APP]`

**BR-INT-06 — Escrow parent exclusivity.**  
`CONSTRAINT escrow_has_one_parent CHECK ((milestone_id IS NOT NULL AND engagement_id IS NULL) OR (milestone_id IS NULL AND engagement_id IS NOT NULL))`. `[DB: CHECK]`

**BR-INT-07 — Partial unique indexes enforce escrow uniqueness per parent.**  
`CREATE UNIQUE INDEX escrow_milestone_unique ON escrow_accounts(milestone_id) WHERE milestone_id IS NOT NULL` and `CREATE UNIQUE INDEX escrow_engagement_unique ON escrow_accounts(engagement_id) WHERE engagement_id IS NOT NULL`. `[DB: UNIQUE INDEX]`

**BR-INT-08 — IPN idempotency index.**  
`CREATE UNIQUE INDEX wallet_tx_idempotency ON wallet_transactions(wallet_id, reference_id) WHERE reference_id IS NOT NULL`. `[DB: UNIQUE INDEX]`

**BR-INT-09 — One review per engagement per reviewer.**  
`UNIQUE (engagement_id, reviewer_id)` on `reviews`. `[DB: UNIQUE]`

**BR-INT-10 — One bid per engagement.**  
`capability_bids.engagement_id UNIQUE`. `[DB: UNIQUE]`

**BR-INT-11 — One wallet per user.**  
`wallets.user_id UNIQUE`. `[DB: UNIQUE]`

**BR-INT-12 — One domain depth row per expert per domain.**  
`UNIQUE (expert_id, domain_code)` on `expert_domain_depths`. `[DB: UNIQUE]`

**BR-INT-13 — One seam claim row per expert per seam.**  
`UNIQUE (expert_id, seam_code)` on `expert_seam_claims`. `[DB: UNIQUE]`

**BR-INT-14 — One message read record per message per user.**  
`UNIQUE (message_id, user_id)` on `message_reads`. `[DB: UNIQUE]`

**BR-INT-15 — Milestone numbers are unique within an engagement.**  
`UNIQUE (engagement_id, milestone_number)` on `milestones`. `[DB: UNIQUE]`

**BR-INT-16 — `platform_settings` is a singleton.**  
The table has exactly one row, seeded at deploy. `platform_wallet_id UNIQUE` ensures no accidental second row creates a second fee destination. `[DB: UNIQUE, seeded]`

**BR-INT-17 — `platform_decisions` is a write-once audit table.**  
No UPDATE or DELETE operations target this table. Every LLM decision event produces a new INSERT. Rows accumulate monotonically. `[APP]`

**BR-INT-18 — All JSON columns use `JSONB` — no `JSON` type.**  
`JSONB` enables indexing and query operators. All structured data arrays and maps (seam lists, artifact bodies, structured signals, stack tags, etc.) are `JSONB`. `[DB]`

**BR-INT-19 — FK dependency order must be respected in migration scripts.**  
The 40-table FK dependency order (as specified in the ER plan Section "FK Dependency Order") must be honoured in Prisma + Alembic migration files: `users → client_profiles / expert_profiles / wallets → wallet_transactions → elicitation_sessions → projects → tech_team_profiles → engagements → capability_bids → milestones → acceptance_criteria → milestone_dod_items → ... → notifications`. `[DB]`

**BR-INT-20 — `virtual_accounts.entity_id` is polymorphic — no FK constraint.**  
`entity_id` holds different UUID types depending on `entity_type` (user_id for WALLET_TOPUP, milestone_id for MILESTONE, engagement_id for SERVICE). No DB-level FK can be declared on polymorphic columns; resolution is application-layer only. `[APP]`

**BR-INT-21 — `platform_decisions.entity_id` is polymorphic — no FK constraint.**  
Same pattern as `virtual_accounts.entity_id`. Resolution of the referenced entity is application-layer only. `[APP]`

**BR-INT-22 — DoD required-item constraint is DB-level — cannot be bypassed by application code.**  
`CONSTRAINT dod_required_cannot_be_na CHECK (NOT (is_required = TRUE AND status = 'NOT_APPLICABLE'))` fires at the PostgreSQL engine level, before any application logic runs. `[DB: CHECK]`

**BR-INT-23 — Wallet balance constraints are DB-level — cannot be bypassed by application code.**  
`CHECK (available_balance >= 0)` and `CHECK (locked_balance >= 0)` on `wallets` fire at the PostgreSQL engine level. `[DB: CHECK]`

**BR-INT-24 — Escrow `amount` and `wallet_transactions.amount` are constrained positive.**  
`escrow_accounts.amount CHECK (amount > 0)` and `wallet_transactions.amount CHECK (amount > 0)`. Zero-value ledger entries are not permitted. `[DB: CHECK]`

**BR-INT-25 — One invitation per project per expert.**  
`UNIQUE (project_id, expert_id)` on `invitations`. Re-inviting an expert resets the existing row via upsert. `[DB: UNIQUE]`

## Cross-Domain Interaction Rules

These rules span multiple domains and are called out explicitly to prevent implementation gaps at integration points.

**BR-XD-01 — Elicitation → Matching chain.**  
`projects.required_seams_json` produced by Stage 5 synthesis directly drives the matching engine's seam alignment scoring (40% weight). Any change to the synthesis output format must be reflected in the matching engine query. `[§0.2]` `[§0.5]`

**BR-XD-02 — Matching → Bid chain.**  
Only experts who appear in the match shortlist for a project may submit a bid. Expert bid submission (`POST /bids`) validates membership in the shortlist before creating the engagement and bid rows. `[APP]`

**BR-XD-03 — Bid SELECTED → Connection → Artifact B chain.**  
`capability_bids.state = 'SELECTED'` → `engagements.state = 'PENDING'` → (both NDA click-throughs) → `engagements.state = 'CONNECTED'` → Artifact B route guard satisfied. Each state transition is a prerequisite for the next. `[APP]`

**BR-XD-04 — Milestone FUNDED → Pay-gated doc release chain.**  
`milestones.state → 'FUNDED'` (via IPN) atomically triggers `paygated_documents.release_state → 'RELEASED'` in the same DB transaction. These two writes cannot be separated. `[DB: atomic TX]`

**BR-XD-05 — DoD gate → Submission → Criteria verification → Approval → Ledger chain.**  
Every step must complete before the next begins: DoD gate (100% required items COMPLETED) → SUBMITTED → all required criteria verified → APPROVED atomic ledger transaction → admin completes withdrawal → RELEASED → (last milestone) → engagement CLOSED → reviews available. `[APP]`

**BR-XD-06 — Dispute → Escrow freeze → Resolution → Milestone APPROVED chain.**  
Filing a dispute (any party) → immediate escrow FROZEN (same TX) → LLM Layer 1 evaluation → AUTO_RESOLVED or MANUAL_REVIEW → ledger distribution → milestone APPROVED. No milestone lifecycle step between DISPUTED and APPROVED can be skipped. `[APP]`

**BR-XD-07 — Subscription state feeds every LLM-calling route guard.**  
`users.subscription_client_tier` and `sub_client_expires_at` are checked at the entry point of every route that calls FastAPI. The subscription middleware is applied uniformly — no route that calls an LLM endpoint is exempt. `[ROUTE]`

**BR-XD-08 — Bank Hub link must precede withdrawal but does not block connection.**  
`engagements.state → 'CONNECTED'` does not require `users.sepay_bank_account_xid IS NOT NULL`. However, `POST /withdrawals` (BR-PAY-12) and admin withdrawal completion both require `bank_account_xid IS NOT NULL`. The system prompts for bank linking at connection but does not block it. `[APP]`

**BR-XD-09 — IPN HMAC verification gates all financial state transitions.**  
No wallet update, escrow creation, milestone state change, or pay-gated document release can occur except through an HMAC-verified SePay IPN. Direct API calls to financial state endpoints (outside the IPN handler) are rejected. `[APP]`

**BR-XD-10 — `platform_decisions` is the single cross-cutting audit record for all AI decisions.**  
Every automated decision by FastAPI (elicitation synthesis, spec auto-return, portfolio evaluation, seam tier upgrade, dispute Layer 1 evaluation, criterion quality gate) writes exactly one `platform_decisions` row. Admin's Platform Integrity Monitor reads exclusively from this table. `[APP]`

**BR-XD-11 — Config CMS → AI Service chain.**  
Admin changes to `prompt_templates` (via `/admin/prompts/:stage`) and `void_code_definitions` (via `/admin/config/void-codes`) are fetched by FastAPI dynamically. Changes take effect within 60 seconds. `[APP]`
